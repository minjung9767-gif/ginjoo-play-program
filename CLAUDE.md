> Claude Code가 새 창(세션)을 열 때마다 이 파일을 자동으로 먼저 읽는다.
> 〔공통〕 부분은 모든 프로젝트가 동일하게 쓰는 규칙이고, 아래는 이 프로젝트(긴주 놀이터)에 맞게 채워둔 부분이다.

---

## 🚨 최우선 규칙 — 무조건 지킬 것  〔공통〕

> **기본값은 "코드를 짜지 않는다"이다.** 요청이 오면 먼저 **민정과 상의·설명·진단만** 한다.
> 코드를 작성·수정하는 것은 **민정이 "코드 짜줘 / 고쳐줘 / 구현해줘"라고 명시적으로 말했을 때만** 한다.
> 좋은 해결책이 보여도 **먼저 제안하고 동의를 받은 뒤** 코드에 손댄다. 시키지 않았는데 미리 고쳐두지 말 것.
> 헷갈리면 "지금 바로 고칠까요, 아니면 먼저 상의만 할까요?"라고 물어본다.

---

## 👤 요청자(민정)와 응대 방식 — 먼저 읽기  〔공통〕

- **민정은 비개발자다.** 전문용어(머지, 디플로이, 브랜치 등)를 쓸 땐 **반드시 쉬운 말이나 비유로 풀어서** 설명한다. 모르는 용어를 던지고 넘어가지 말 것.
- 답변은 **한국어로, 간결하고 친절하게.** 길게 늘어놓기보다 핵심 → 다음 할 일 순으로.
- **모바일(폰)에서 작업하는 경우가 많다.** 맥북 없이 `claude.ai/code` + GitHub 앱으로 진행한다는 전제.
- **애매하거나 구조에 영향 있는 변경은 먼저 물어본다** (`AskUserQuestion`). 임의로 큰 결정 내리지 말 것.
- (위 "최우선 규칙" 참고) 기본은 상의·진단이고, **명시적으로 시킬 때만** 코드를 짠다.
- 토큰 비용·속도에 민감하다. 전체를 읽지 말고 아래 "코드 지도"로 좁혀서 접근한다.
- 작업이 끝나면 **무엇을 했는지 + 다음에 사용자가 할 일**을 분명히 알려준다.

## 🛠 작업 규칙  〔공통〕

1. **항상 새 작업 브랜치에서** 작업한다 (`main`에 직접 커밋하지 말 것). 브랜치명 예: `claude/<기능요약>`.
2. 변경 후 **커밋 → 푸시 → PR 생성**. PR 본문은 비개발자도 이해할 한국어로 "무엇을/왜" 정리.
3. **이슈 없으면 머지까지 자동 진행.** 구현을 지시받았으면 **커밋 → PR → CI 통과 확인 → 이슈 없으면 머지(squash)까지** 한 번에 진행하고 결과를 알린다.
   - 머지 전 **CI 초록(success) 확인** (CI가 없으면 직접 빌드/실행으로 확인).
   - **이슈가 있으면 머지하지 말고 먼저 보고**한다(CI 실패·충돌·테스트 깨짐·애매한 결정 등). 쉬운 말로 설명하고 상의한다.
   - "최우선 규칙"은 그대로 — 안 시키면 코드 짜서 밀어붙이지 말 것.
4. **규칙 변경 요청도 처리한다.** "이 규칙 추가해줘 / 앞으로 이렇게 해줘"라고 하면 **이 `CLAUDE.md`에 반영**하고 PR → CI → 이슈 없으면 머지까지 진행한다.
   - 그 규칙이 모든 프로젝트 공통이면, 민정에게 "이건 `project-starter`(규칙 원본)에도 반영해두면 다음 프로젝트부터 자동 적용돼요"라고 알려준다.
5. 코드 작업 모델은 **Opus 계열** 권장.

## 📦 이 프로젝트가 뭔지

**긴주 놀이터 (ginjoo-play-program)** — 22개월 아기를 위한 **카메라 기반 웹 놀이**. 설치 없이 브라우저에서 바로 실행되고, 아기 영상은 기기 밖으로 나가지 않고 브라우저 안에서만 처리된다(개인용).

- **스택:** 순수 HTML / CSS / JavaScript, **빌드 과정 없음**.
- 얼굴 인식: Google MediaPipe Tasks Vision (FaceLandmarker) — CDN에서 로드.
- 스티커는 이미지 파일 없이 **이모지 + 캔버스 그림**으로 구현.
- 배경 음악/효과음은 **Web Audio API**로 합성(음원 파일 없음).
- 현재 완성된 놀이: **🪞 매직 거울**. (움직임 마법·까꿍 동물은 준비 중)

## 🚀 배포 구조

> 코드를 고치면 실제로 어디에 반영되는지: **GitHub Pages 자동 배포**.

```
코드 수정 → 작업 브랜치 → PR → main에 머지 → GitHub Pages 자동 배포 → 실제 반영
```

- 배포 설정 파일: `.github/workflows/deploy-pages.yml` (`main` 브랜치에 푸시될 때마다 자동 배포).
- 공개 주소: **https://minjung9767-gif.github.io/ginjoo-play-program/**
- main에 머지된 뒤 GitHub가 사이트를 새로 만드는 데 **1~2분** 걸린다. 진행 상황은 저장소 **Actions** 탭에서 확인.

## 🧪 테스트 / 확인 방법

- **카메라는 `file://`(파일 더블클릭)로 열면 동작하지 않는다.** 반드시 `https` 또는 `localhost`로 띄워서 확인.
- 로컬 확인: 프로젝트 폴더에서 아래 실행 후 브라우저로 `http://localhost:8000` 접속.
  ```bash
  python3 -m http.server 8000   # 파이썬이 없으면: npx serve
  ```
- 빌드/자동 테스트는 없다. CI는 **GitHub Pages 배포 워크플로뿐**(빌드·테스트 단계 없음). 그래서 변경 후엔 **직접 브라우저로 띄워 동작을 눈으로 확인**한다.

## ⚠️ 알아둘 함정 / 주의사항

- **카메라는 https/localhost에서만 동작.** `file://`로는 안 됨. 로컬에서 더블클릭으로 확인하려다 "카메라가 안 켜진다"고 오해하기 쉽다.
- **빌드 과정이 없다.** 파일을 고치면 그게 곧 사이트 내용이 된다(트랜스파일/번들 없음).
- **인터넷 연결 필요.** 첫 실행 시 얼굴 인식 모델을 다운로드(이후 캐시됨).
- **자동 테스트(CI 검사)가 없다.** 배포 워크플로만 있으므로, 머지 전엔 직접 실행해 확인한다. 필요하면 검사 추가를 제안할 것.
- 권장 브라우저: 크롬(Chrome), 엣지(Edge). 기준 기기는 삼성 노트북(가로·터치).
